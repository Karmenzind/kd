# kd 终端交互后续计划：过渡、完成状态与宽度自适应

本计划承接此前的终端交互改造，只处理当前实现中仍有实际价值的缺口。已经落地的进度渲染基础、TTY/机器输出隔离、延迟显示、Unicode/ASCII 降级、查询词 shimmer、真实阶段更新和并发生命周期保护不再重复实施。

## 当前实现基线

当前 `internal/ui` 已经具备：

- 独立的 `Progress`/no-op 实现，业务代码不直接操作 ANSI；
- 约 150ms 的延迟显示和约 90ms 的低频刷新，快速查询不闪烁；
- `starting`、`local`、`remote`、`format` 四个真实阶段；
- 查询词前景色 shimmer、Unicode spinner、ASCII fallback 和无动画模式；
- 状态写入 stderr、最终结果写入 stdout，JSON、日志流、非 TTY 输出不受污染；
- `Stop` 幂等、并发 `Update`/`Stop` 安全、context 取消和 writer 错误退出；
- 基础终端能力检测及 Windows 保守降级；
- daemon 启动、在线查询和结果格式化阶段的接入与回归测试。

因此，后续不再重做渲染架构，也不引入 TUI 框架、主题系统、新配置项或第三方依赖。

## 本轮范围

按维护价值和风险排序实施以下三项：

1. 终端宽度自适应；
2. 轻量、可中断的阶段切换；
3. 仅对已经显示过的进度提供极短完成状态。

三项都必须保持“查询完成优先”：动画不得形成等待队列，也不得为了播放完整效果延迟快速查询或正式结果。

## 一、终端宽度自适应

### 目标

替代当前固定 rune 数截断，使状态行不会在窄终端换行，并在宽终端保留查询词和阶段信息。

建议分级渲染：

- 宽度充足：`<spinner> <query> · <phase>`；
- 宽度有限：缩短查询词，仍尽量保留阶段；
- 很窄：`<spinner> <query>`；
- 极窄或宽度未知且风险较高：只显示 spinner，或沿用当前保守 fallback。

### 实现约束

- 终端宽度探测集中在 `internal/ui`，通过可注入函数或 option 测试，不把 `term.GetSize` 散落到渲染逻辑；
- 仅对真实终端文件探测宽度，探测失败时使用保守默认值；
- 每次绘制前可以读取最新宽度，以支持窗口缩放，但不得在每帧重复探测环境能力；
- 继续按 rune 处理 Unicode。没有可靠列宽库时采用保守估算，不为精确 East Asian Width 引入依赖；
- Basic 模式不做宽字符逐字 shimmer，避免旧终端残影；
- 所有宽度计算必须防止负数、越界、空字符串 panic 和截断后 ANSI reset 缺失。

## 二、轻量阶段切换

### 目标

让已经可见的状态从 `local` 切换到 `remote` 或 `format` 时更连贯，但不伪造进度。

第一版只实现一种简单效果：阶段分隔点短暂变化或 spinner 短暂收束，然后直接显示新阶段。不要实现字符 morph、编辑距离或多行动画。

### 行为规则

- 只有进度行已经显示时才播放过渡；延迟窗口内的阶段变化直接合并为最新状态；
- 过渡总时长控制在约 120ms，帧间隔复用现有刷新节奏；
- 连续快速 `Update` 只保留最新阶段，不排队播放历史阶段；
- `Stop`、context 取消、writer 错误和查询完成可立即中断过渡；
- Basic 模式直接切换阶段，Plain/no-op 模式无任何输出；
- 未知阶段继续安全显示文本，不进入特殊过渡分支。

## 三、短暂完成状态

### 目标

当慢查询已经显示过动态状态时，提供一个极短的成功或失败反馈，然后清行并交回现有结果/错误输出。

### API 与集成

先评估以最小方式扩展 `Progress`，例如增加明确的完成方法和简单 outcome 类型。接口应同时更新 no-op 实现，且 outcome 只表达视觉上的成功或失败，不携带业务错误文本。

CLI 查询入口负责在已知最终结果后通知渲染器：

- 成功：短暂显示简单完成标记，随后清行并输出原有结果；
- 失败：短暂显示简单失败标记，随后清行并走现有正式错误处理；
- 快速查询、未显示过进度、非 TTY 或机器输出：完全跳过完成动画。

完成状态最多约 80ms，并且必须可中断。若实现会明显延迟结果或增加复杂同步，则退化为直接 `Stop`，不强行保留视觉效果。

### 输出边界

- 正式结果和正式错误的措辞、格式、颜色与退出行为保持不变；
- 渲染器不得格式化业务错误，也不得向 stdout 写状态；
- 不实现“结果标题接管状态行”。当前状态在 stderr、结果在 stdout，强行原地接管会破坏重定向语义并增加跨终端不确定性；
- 不隐藏光标。

## 测试计划

继续使用 Go 标准库测试，不依赖真实用户终端或固定时序。必要时注入宽度读取器、ticker/时钟边界或极短可控 duration。

### 宽度与渲染

- 宽终端包含查询词和阶段；
- 窄终端按预期依次降级，且不换行或留下旧帧；
- 极小、零、负值和探测失败不会 panic；
- Unicode、中文、空查询和超长查询均能安全截断；
- resize 后下一帧使用新宽度；
- Basic 不输出 shimmer/Braille/不受支持的 ANSI，Enhanced 保留当前效果。

### 阶段过渡

- 进度尚未显示时阶段更新被合并，不产生输出；
- 已显示状态的阶段变化触发一次短过渡；
- 快速连续更新不形成过渡积压，最终显示最新阶段；
- `Stop`、取消和 writer 错误立即终止过渡；
- 并发更新与停止在 race detector 下无竞争。

### 完成状态

- 仅在进度已经可见时显示成功或失败状态；
- 快速完成、Plain、no-op 和机器输出无额外内容；
- 完成状态时长有上限，结束后状态行被清理；
- 失败标记不替代、吞掉或改写正式错误；
- stdout 只包含最终结果，stderr 只包含临时状态和原有错误输出；
- 重复完成/停止安全，不产生额外帧或 goroutine 泄漏。

## 实施顺序

1. 为当前渲染路径补充宽度读取注入点和纯渲染测试；
2. 实现宽度分级与保守截断，先稳定跨平台输出；
3. 将阶段变化建模为“当前状态 + 最新目标状态”，实现可中断、无队列的短过渡；
4. 以最小 API 扩展完成通知，并仅在 CLI 查询边界接入；
5. 补充生命周期、输出隔离和并发回归测试；
6. 检查 Windows Basic/Enhanced、Unix TTY、管道和重定向行为；
7. 运行格式化、静态检查、完整测试、race detector 和纯 Go 跨平台构建。

## 明确不做

以下内容来自原计划的实验设想，但当前收益不足以抵消维护和兼容成本，本计划继续不实施：

- 背景色 sweep、多字符残影和复杂 morph；
- marquee 进度条或虚假百分比；
- 结果释义逐字/分块展开；
- terminal bell、OSC 或系统通知；
- 多行 TUI、主题或插件系统；
- 为动画增加配置文件字段或顶层 CLI 参数；
- 改变 stdout/stderr 分工、查询协议或核心查词流程。

## 验证命令

```powershell
pwsh -NoProfile -Command "gofmt -w internal/ui/*.go cmd/kd/*.go"
pwsh -NoProfile -Command "go test -count=1 ./..."
pwsh -NoProfile -Command "go vet ./..."
pwsh -NoProfile -Command "go test -race -count=1 ./..."
pwsh -NoProfile -Command "$env:CGO_ENABLED='0'; go build -o build/kd-ui-test.exe ./cmd/kd"
```

如果修改终端平台文件，还应使用 `CGO_ENABLED=0` 对 Linux、macOS 和 Windows 目标做编译检查。验证后删除本地构建产物，不提交 `build/kd-ui-test.exe`。

## 完成标准

- 状态行能根据终端宽度稳定降级，不因长文本或宽字符破坏下一行；
- 阶段变化连贯但不排队、不伪造进度、不延迟查询完成；
- 完成状态只服务于已经可见的慢查询，快速查询仍为零额外可见输出；
- stdout、stderr、JSON、管道、重定向和现有结果格式保持兼容；
- Plain、Basic、Enhanced 的既有降级规则继续有效；
- 无 goroutine、ticker、ANSI 或状态行残留；
- Windows、macOS、Linux 构建可用，且 `go test ./...`、`go vet ./...`、`go test -race ./...` 通过；
- 不新增第三方依赖，不扩大为完整 TUI 或输出架构重构。
