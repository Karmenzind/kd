请在 `kd` 项目的 release build 流程中增加 UPX 自动压缩。

本任务的目标是减小 Windows 和 Linux 发布二进制体积，同时确保：

* macOS 构建不使用 UPX；
* 本地普通开发构建不强制依赖 UPX；
* release 构建可以显式关闭 UPX；
* UPX 压缩失败时不会悄悄发布损坏产物；
* 压缩后的二进制必须经过基本运行验证；
* 保留清晰的构建日志和压缩前后体积数据。

## 总体策略

UPX 只用于以下目标：

```text
windows/amd64
windows/386
linux/amd64
linux/arm64
```

如果当前项目没有构建其中某些目标，则只处理实际已有目标。

以下目标不使用 UPX：

```text
darwin/amd64
darwin/arm64
```

也不要对以下产物使用 UPX：

* source archive；
* completion script；
* 配置文件；
* checksum 文件；
* 已经归档后的 zip/tar.gz；
* macOS Mach-O；
* 非可执行文件。

UPX 必须在：

```text
go build
→ strip/debug info 处理
→ UPX
→ smoke test
→ archive
→ checksum
```

这个顺序中执行。

不要对已经生成的 zip 或 tar.gz 使用 UPX。

## 第一阶段：审查现有构建流程

先定位当前所有构建入口，包括但不限于：

* `Makefile`；
* `Taskfile.yml`；
* `justfile`；
* `scripts/`；
* GitHub Actions；
* GoReleaser；
* release shell script；
* PowerShell build script；
* 当前 release archive 和 checksum 生成逻辑。

输出简短审查结果，说明：

1. 当前支持哪些 GOOS/GOARCH；
2. release 构建从哪里触发；
3. 是否使用 GoReleaser；
4. 是否已经使用 `-s -w`；
5. 是否生成 zip/tar.gz；
6. checksum 在哪个阶段生成；
7. 是否已经有 smoke test；
8. Windows 和 Linux 构建是在什么 runner 上完成；
9. 当前是否保留未压缩二进制。

根据仓库现有风格修改，不要平行创建第二套完全不同的构建系统。

## 第二阶段：构建参数

确保 release 构建使用合理的 Go linker flags，例如项目当前已有时保持现状：

```text
-ldflags="-s -w"
```

如果项目还会注入：

* version；
* commit；
* build date；

应合并现有 ldflags，不要覆盖。

例如概念上：

```text
-s -w -X main.version=... -X main.commit=...
```

不要在本任务中改变版本注入结构。

不要使用会破坏功能或调试能力的实验性编译参数。

## 第三阶段：UPX 版本固定

在 CI 或 release 配置中固定 UPX 版本，不要使用未经固定的 `latest`。

建议当前固定为：

```text
5.2.0
```

将版本集中定义，例如：

```text
UPX_VERSION=5.2.0
```

或构建系统中的等价字段。

不要在多个脚本中分别硬编码版本。

下载 UPX 时：

* 从 UPX 官方 GitHub release 获取；
* 根据 runner OS 和 architecture 选择正确包；
* 校验官方发布的 SHA-256；
* 不从不明镜像下载；
* 不把 UPX 二进制提交进仓库；
* 可合理使用 CI cache，但 cache key 必须包含 UPX 版本。

如果现有 CI 有标准工具安装机制，应复用现有机制。

## 第四阶段：可配置开关

增加一个统一开关：

```text
UPX_ENABLED
```

推荐语义：

```text
UPX_ENABLED=1
UPX_ENABLED=true
```

表示启用。

以下值表示关闭：

```text
UPX_ENABLED=0
UPX_ENABLED=false
```

默认策略：

* 正式 release pipeline：默认启用；
* 本地普通 `go build`：不启用；
* 本地 release build：有 UPX 时启用或通过显式开关启用；
* macOS target：无论开关如何都跳过。

不要让普通开发者执行：

```text
go build ./...
```

时必须预装 UPX。

如果现有项目偏好显式行为，也可以让 release pipeline 明确设置：

```text
UPX_ENABLED=1
```

而脚本默认关闭。

最终采用哪种方式需与现有构建风格一致，但必须保证普通开发构建不受影响。

## 第五阶段：UPX 参数

默认使用：

```text
upx --best <binary>
```

暂时不要默认使用：

```text
--ultra-brute
--brute
```

原因：

* 构建时间明显增加；
* 收益不稳定；
* CI release 时间增加；
* 可能引入额外兼容风险。

如果压缩耗时很短且现有构建验证充分，可以评估：

```text
upx --best --lzma
```

但必须先比较实际体积和启动表现。

优先从：

```text
upx --best
```

开始。

Windows 示例：

```text
upx --best dist/kd-windows-amd64.exe
```

Linux 示例：

```text
upx --best dist/kd-linux-amd64
```

不要对 macOS 文件执行。

## 第六阶段：压缩前验证

UPX 执行前检查：

* 文件存在；
* 文件大小大于零；
* 目标是 Windows PE 或 Linux ELF；
* 文件尚未被 UPX 压缩；
* 当前 target 在允许列表中；
* UPX 命令可用；
* UPX 版本符合构建配置。

可以使用：

```text
upx --version
upx -t <binary>
```

但注意未压缩文件执行 `upx -t` 的返回行为，不能把“尚未压缩”误判为损坏。

如果脚本重复运行：

* 不要重复压缩同一文件；
* 可以通过 `upx -l` 或等价方式检测；
* 或确保每次从干净的 dist 目录重新构建。

优先采用干净构建目录，减少状态判断复杂度。

## 第七阶段：保留原始体积信息

压缩前记录：

```text
original_size
```

压缩后记录：

```text
compressed_size
saved_bytes
saved_percent
```

日志示例：

```text
UPX windows/amd64:
  original:   18.4 MiB
  compressed: 6.7 MiB
  saved:      63.6%
```

跨平台脚本不要依赖 GNU 特有的 `stat` 参数。

如果构建逻辑使用 Go、Python 或 PowerShell，可以通过标准文件 API 获取大小。

不要为了打印体积引入新的大型依赖。

## 第八阶段：UPX 完整性验证

压缩完成后必须执行：

```text
upx -t <binary>
```

如果测试失败：

* release build 失败；
* 不继续归档；
* 不生成 checksum；
* 不上传该产物。

不能只检查 `upx` 命令退出码而不执行压缩后测试。

## 第九阶段：二进制 smoke test

UPX 压缩后，对能够在当前 runner 上运行的二进制执行：

```text
kd --version
kd --help
```

验证：

* 退出码为 0；
* 不崩溃；
* 输出包含程序名称或版本；
* 不创建异常 daemon；
* 不访问网络；
* 不等待 stdin；
* 不显示交互动画。

例如：

Linux runner 验证 Linux binary：

```text
./kd-linux-amd64 --version
```

Windows runner 验证 Windows binary：

```powershell
.\kd-windows-amd64.exe --version
```

交叉编译且不能在当前 runner 运行的产物：

* 仍执行 `upx -t`；
* 不伪装成已完成运行测试；
* 在构建日志中明确标记 `runtime smoke test skipped`。

如果 CI matrix 可以在对应 OS 上原生构建和测试，优先使用原生 runner。

## 第十阶段：功能 smoke test

如果能够不依赖网络和用户配置，增加一个最小功能测试。

例如：

* `--version`；
* `--help`；
* 读取内置或临时配置；
* 打开临时 SQLite；
* 执行不会请求网络的命令。

不要在 release pipeline 中依赖在线词典服务来判断 UPX 是否成功。

网络波动不能导致发布随机失败。

## 第十一阶段：归档顺序

压缩和测试完成后再归档：

Windows：

```text
kd-windows-amd64.exe
→ UPX
→ smoke test
→ zip
```

Linux：

```text
kd-linux-amd64
→ UPX
→ smoke test
→ tar.gz
```

macOS：

```text
kd-darwin-arm64
→ 不执行 UPX
→ smoke test
→ tar.gz
```

checksum 必须针对最终归档或最终发布文件生成。

不要在 UPX 前生成 checksum。

## 第十二阶段：macOS 明确跳过

macOS 构建必须明确输出：

```text
UPX skipped for darwin/arm64
```

或等价日志。

不要：

* 尝试用 Linux UPX 处理 Mach-O；
* 在 macOS runner 上临时安装第三方非官方 UPX fork；
* 压缩后再 codesign；
* codesign 后再压缩；
* 破坏 notarization；
* 因为 macOS 无法压缩而让整个 release 失败。

macOS 继续只使用：

```text
go build -ldflags="-s -w ..."
```

如果未来增加 codesign/notarization，本任务不能干扰该顺序。

## 第十三阶段：Windows Defender 和误报风险

UPX 压缩后的 Windows 二进制可能更容易触发安全软件启发式检测。

因此构建流程需要：

* 提供关闭 UPX 的开关；
* 在 release 文档中说明 Windows 产物经过 UPX 压缩；
* 保留构建时的 UPX 版本；
* 建议未来对 Windows release 做 VirusTotal 或 Defender 检查；
* 不把安全软件误报简单视为用户环境问题。

本任务不要求上传二进制到第三方扫描服务，除非仓库已有安全扫描流程。

不要把 API key 或敏感 token 写入脚本。

## 第十四阶段：是否保留未压缩产物

优先考虑以下设计：

### 发布包

发布 UPX 压缩后的二进制：

```text
kd-windows-amd64.zip
kd-linux-amd64.tar.gz
```

### CI artifact

可选保留未压缩二进制作为短期 CI artifact：

```text
kd-windows-amd64-unpacked.exe
kd-linux-amd64-unpacked
```

用途：

* 比较误报；
* 调试 UPX 问题；
* 分析崩溃；
* 比较启动性能。

不要求把未压缩版本同时放到正式 GitHub Release，除非当前发布策略确有需要。

如果 CI 存储成本不值得，可以不长期保留，但构建流程必须可以通过：

```text
UPX_ENABLED=0
```

重新生成未压缩产物。

## 第十五阶段：压缩收益阈值

可以增加保护逻辑：

如果压缩后节省比例低于合理阈值，例如：

```text
5%
```

则：

* 输出 warning；
* 仍可继续发布；
* 不必自动失败。

不要因为某个平台压缩收益较低而直接破坏 release。

如果 UPX 生成文件大于原文件：

* 恢复原文件；
* 输出 warning；
* 发布未压缩版本。

为此可以先复制为临时文件：

```text
binary
binary.unpacked
```

确认 UPX 成功后再决定是否保留压缩结果。

避免 UPX 失败后丢失原始二进制。

## 第十六阶段：失败策略

正式 release 构建中，当：

```text
UPX_ENABLED=1
```

且目标是 Windows/Linux 时：

以下情况应使对应 release job 失败：

* UPX 下载失败；
* checksum 校验失败；
* UPX 执行失败；
* `upx -t` 失败；
* native smoke test 失败；
* 压缩后文件为空；
* 压缩后文件无法执行。

不要静默 fallback 到未压缩二进制，否则发布者可能误以为 release 已经压缩。

只有明确配置：

```text
UPX_OPTIONAL=1
```

时才允许失败后回退。

第一版不一定需要 `UPX_OPTIONAL`。更简单的策略是：

```text
UPX_ENABLED=1 → 失败即失败
UPX_ENABLED=0 → 完全跳过
```

## 第十七阶段：本地构建体验

如果仓库已有 build script，支持类似：

PowerShell：

```powershell
$env:UPX_ENABLED = "1"
./scripts/build.ps1
```

Unix：

```bash
UPX_ENABLED=1 ./scripts/build.sh
```

也可以通过 Makefile：

```text
make release UPX=1
```

但不要同时发明多个不同开关。

选择一个项目现有风格一致的入口。

如果 UPX 未安装：

* 普通 build 不受影响；
* release 且 UPX 已启用时给出清晰错误；
* 错误应包含安装提示；
* 不自动使用管理员权限安装软件。

Windows 本地安装提示优先使用：

```powershell
winget install UPX.UPX
```

但 CI 应使用固定版本下载，而不是无版本约束的 winget 安装。

## 第十八阶段：测试

增加构建逻辑测试，至少覆盖：

* Windows target 会选择 UPX；
* Linux target 会选择 UPX；
* Darwin target 明确跳过；
* `UPX_ENABLED=0` 跳过；
* UPX 不存在时返回清晰错误；
* UPX 返回非零时构建失败；
* `upx -t` 失败时构建失败；
* smoke test 失败时构建失败；
* 压缩前后体积正确记录；
* 不对 archive 执行 UPX；
* checksum 在 UPX 后生成；
* 重复执行不会重复压缩；
* 路径包含空格时仍然正常；
* Windows `.exe` 文件名处理正确。

如果构建逻辑是 shell script，可以通过 fake `upx` executable 测试，而不是在单元测试中真实压缩。

如果构建逻辑是 Go 程序，使用依赖注入或可替换 command runner。

## 第十九阶段：实际验证

至少用一个 Windows binary 和一个 Linux binary验证：

```text
go build -ldflags="-s -w" ...
upx --best ...
upx -t ...
```

记录：

* 压缩前大小；
* 压缩后大小；
* 压缩比例；
* `--version` 是否正常；
* `--help` 是否正常；
* 启动是否有明显异常。

同时验证 macOS 构建未经过 UPX。

执行项目现有测试：

```text
go test ./...
go test -race ./...
go vet ./...
```

如果 release 配置支持 dry-run，也执行 dry-run。

## 不允许执行的事项

本任务禁止：

* 对 macOS 使用 UPX；
* 对所有平台无条件执行 UPX；
* 修改 Go 业务逻辑；
* 修改 CLI 参数；
* 修改输出格式；
* 修改 SQLite schema；
* 引入 CGO；
* 使用未经固定的 UPX latest；
* 从非官方来源下载 UPX；
* 在 checksum 生成后修改二进制；
* 忽略 `upx -t`；
* UPX 失败后静默继续；
* 使用 `--brute` 或 `--ultra-brute` 作为默认参数；
* 把 UPX 二进制提交到 Git；
* 把未压缩文件误放入正式压缩发布包。

## 推荐提交

建议单独一个 commit：

```text
build(release): compress Windows and Linux binaries with UPX
```

如果测试或构建框架改动较多，可以拆成：

```text
test(build): cover release binary post-processing
build(release): add optional UPX compression
```

不要与依赖升级、CLI completion 或 renderer 修改混在同一个 commit。

## 完成标准

完成后必须满足：

* Windows release binary 使用 UPX；
* Linux release binary 使用 UPX；
* macOS release binary 明确跳过；
* UPX 版本固定；
* 下载文件校验 checksum；
* 普通开发构建不依赖 UPX；
* 可以显式关闭 UPX；
* UPX 在归档和 checksum 之前执行；
* 压缩后执行 `upx -t`；
* 可原生运行的产物执行 smoke test；
* 日志显示压缩前后体积；
* UPX 失败不会发布损坏产物；
* 不改变用户可见功能；
* 项目测试通过；
* release dry-run 或等价验证通过。

5. UPX 使用的固定版本；
6. 使用了哪些 UPX 参数；
7. 压缩前后体积；
8. 是否执行 `upx -t`；

10. 如何关闭 UPX；
11. UPX 失败时如何处理；
12. 是否保留未压缩产物；

不要在完成后继续修改其他 release 行为。
